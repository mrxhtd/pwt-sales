System Architecture

PWT Sales Tracker

A single-page application for PWT International's sales team to manage leads, clients, and water treatment product inventory.

01 — The Big Picture
How Everything Connects

The app is a static SPA served by Vercel, backed by serverless API functions and a Supabase PostgreSQL database. Here's the request lifecycle:

🌐 Browser
User opens index.html
JS calls fetch('/api/auth')
Cookie pwt_auth sent automatically
Renders UI from JSON responses
→
⚡ Vercel Functions
/api/auth.js — login, logout, check
/api/sites.js — leads CRUD
/api/clients.js — clients CRUD
/api/products.js — inventory CRUD
/api/engineers.js — team mgmt
/api/convert.js — lead → client
/api/extract.js — AI extraction
→
🗄️ Supabase
engineers — user accounts
sessions — auth tokens
sites — sales leads
clients — converted clients
client_products — inventory
Key Insight

There is no client-side Supabase SDK. The browser never talks to the database directly. Every request goes through Vercel serverless functions, which authenticate the user, enforce ownership rules, and then query Supabase using the service role key (full access). This means all authorization logic lives server-side in the /api/ layer.

02 — Tech Stack
What Powers the App
📄
Frontend

A single index.html file (3,800+ lines) containing all HTML, CSS, and vanilla JavaScript. No frameworks, no build step, no bundler. Served as a static file by Vercel's CDN.

⚡
Backend

7 Vercel serverless functions in /api/, each a single ES module exporting a default handler(req, res). Node.js runtime, no Express — just raw request/response handlers.

🗄️
Database

Supabase PostgreSQL with 5 tables. Accessed via the @supabase/supabase-js client using the service role key. RLS is enabled but permissive — authorization is handled in API code.

🔐
Auth

Custom session-based auth. Passwords hashed with bcryptjs. Sessions stored in the database as UUID tokens. Delivered via HttpOnly cookies with 30-day expiry.

🤖
AI Integration

Gemini 2.5 Flash API for natural language extraction. Engineers describe a lead in text, voice, or photo — the AI parses it into structured form fields automatically.

🚀
Deployment

Vercel with zero config. Static files served from CDN, API routes as serverless functions. Security headers set via vercel.json. Environment variables managed in Vercel dashboard.

03 — Authentication
How Users Get In

Custom session-based auth — no Supabase Auth, no JWTs, no third-party auth providers.

Login Flow

1. User submits credentials
Browser sends POST /api/auth with { username, password }. The username is lowercased and trimmed.
2. Rate limit check
Server checks in-memory map: max 7 attempts per 15 minutes per IP+username combo. Returns 429 if exceeded.
3. Lookup & verify
Query engineers table by username. Check is_active. Compare password with bcrypt hash. Same generic error for "not found" and "wrong password" to prevent user enumeration.
4. Create session
Generate crypto.randomUUID() token. Insert into sessions table with 30-day expiry. Set pwt_auth cookie: HttpOnly; Secure; SameSite=Lax; Path=/.
5. Return engineer info
Response: { ok: true, engineer: { id, fullName, role } }. Frontend stores this in currentEngineer and shows the app shell.

Session Verification (every API call)

// Every protected endpoint starts with:
const session = await getSession(req);
if (!session) return res.status(401).json({ error: 'Unauthorized' });

// getSession() does:
// 1. Parse pwt_auth cookie from request headers
// 2. Query: sessions JOIN engineers WHERE token = ?
// 3. Check expires_at > now()
// 4. Check engineers.is_active = true
// 5. Return { engineerId, fullName, role } or null
Session Guard

When the user switches back to the browser tab, a visibilitychange listener calls GET /api/auth to re-verify the session is still valid. If expired, the user is kicked back to login.

Expired Session Cleanup

A background job in getSession() deletes expired sessions from the DB — runs at most once per hour per function instance to avoid hammering the database.

04 — Data Model
Database Schema & Relationships

5 PostgreSQL tables in Supabase. All primary keys are text UUIDs. Foreign keys enforce referential integrity with cascading deletes.

👤 engineers
PK id TEXT
username TEXT UNIQUE
password TEXT (bcrypt)
full_name TEXT
role 'admin' | 'engineer'
is_active BOOLEAN
🔑 sessions
PK token TEXT (UUID)
FK engineer_id → engineers.id
created_at TIMESTAMPTZ
expires_at TIMESTAMPTZ
📍 sites Leads
PK id TEXT
FK engineer_id → engineers.id
name TEXT
contact, phone, location TEXT
equipment, specs TEXT
status TEXT (pipeline stage)
next_action, due_date, notes TEXT
🏢 clients Clients
PK id TEXT
FK engineer_id → engineers.id
FK converted_from → sites.id (nullable)
name, contact, phone, location TEXT
equipment, specs, notes TEXT
converted_at TIMESTAMPTZ
📦 client_products Inventory
PK id TEXT
FK client_id → clients.id
category boilers | cooling_towers | chillers | swimming_pools
product_name TEXT (from catalog)
model, quantity TEXT, INTEGER
install_date, next_maintenance_date DATE
status 'active' | 'running_out_of_stock'

Relationship Map

engineers ──┬── 1:N ──→ sessions (auth tokens, CASCADE delete)
├── 1:N ──→ sites (leads owned by engineer)
└── 1:N ──→ clients (clients owned by engineer, CASCADE delete)

sites ────── 1:1 ──→ clients.converted_from (a client can come from a lead)

clients ──── 1:N ──→ client_products (products purchased, CASCADE delete)
05 — API Layer
Serverless Endpoints

Each file in /api/ becomes a Vercel serverless function. Every endpoint (except extract) follows the same pattern: authenticate → check ownership → query Supabase → return JSON.

Endpoint Methods Auth Purpose
/api/auth GET POST DELETE Public (POST)
Cookie (GET/DEL)		 GET: check session validity → return engineer info
POST: login with username+password → set cookie
DELETE: logout → clear cookie & delete session
/api/sites GET POST DELETE Session required GET: list leads (engineer's own, or all if admin)
POST: create/update lead (upsert, ownership check)
DELETE: remove lead (ownership check)
/api/clients GET POST DELETE Session required GET: list clients with product counts & low-stock counts
POST: create/update client (ownership check)
DELETE: remove client (cascades to products)
/api/products GET POST DELETE Session required GET: list products for a client (verifies client ownership)
POST: create/update product (verifies client ownership)
DELETE: remove product (verifies via join to client)
/api/convert POST Session required Convert a "Closed Won" lead into a client. Copies all fields, sets converted_from reference. Prevents duplicate conversion.
/api/engineers GET POST DELETE Admin only GET: list all engineers
POST: create/update engineer (bcrypt hash on create)
DELETE: soft-disable (sets is_active=false)
/api/extract POST Session required Send text/image/audio to Gemini 2.5 Flash → returns structured lead data (name, contact, equipment, status, etc.)

Common Pattern in Every Endpoint

export default async function handler(req, res) {
  // 1. Authenticate
  const session = await getSession(req);
  if (!session) return res.status(401);

  // 2. Extract role info
  const { engineerId, role } = session;
  const isAdmin = role === 'admin';

  // 3. Route by HTTP method
  if (req.method === 'GET') { /* query with ownership filter */ }
  if (req.method === 'POST') { /* upsert with ownership check */ }
  if (req.method === 'DELETE') { /* delete with ownership check */ }
}
06 — Frontend Architecture
The Single-Page Application

Everything lives in one index.html file. No React, no Vue — just vanilla JS manipulating the DOM directly. The file has a clear internal structure:

🎨
CSS (Lines 1–1493)

~1,500 lines of embedded styles. CSS custom properties for theming (dark/light). Fully responsive with breakpoints at 900px, 768px, 600px, 480px, 380px, and 320px. No external CSS framework.

🏗️
HTML (Lines 1494–1944)

App shell with splash screen, login overlay, header, tab bar, three main sections (leads, clients, engineers), a shared detail page, modals for add/edit, preview overlay, and toast notifications.

⚙️
JavaScript (Lines 1946–3820)

~1,900 lines of vanilla JS. Global state variables, fetch wrappers, DOM rendering functions, event handlers, hash router, and utility helpers. No modules — everything is in a single <script> tag.

📊
Rendering Pattern

Data → innerHTML. Functions like renderTable(), renderClientTable(), buildCustomerDetailHTML() build HTML strings from arrays and inject them. All user data is escaped with esc().

08 — Roles & Access Control
Who Can Do What

Two roles: admin and engineer. The role is stored in the engineers.role column and checked both server-side and client-side.

Admin Full Access

  • See all leads from all engineers
  • See all clients from all engineers
  • Edit/delete any lead or client
  • View the Team tab (engineer list)
  • Create new engineer accounts
  • Disable/edit engineer accounts
  • View engineer profiles (their leads & clients)
  • Filter leads/clients by specific engineer

Engineer Scoped Access

  • See only their own leads
  • See only their own clients
  • Create/edit/delete own leads & clients
  • Manage products on own clients
  • Convert own "Closed Won" leads to clients
  • Use AI extraction to create leads
  • Cannot see Team tab
  • Cannot see other engineers' data

How It's Enforced

Server-side (API layer)

Every GET query filters by engineer_id unless the user is admin. Every POST/DELETE verifies the record belongs to the requesting engineer. Admin-only endpoints return 403 for non-admins.

Client-side (UI layer)

Elements with class .admin-only are hidden via display:none for non-admins. The #engineer/ and #engineers routes redirect non-admins to #. This is cosmetic — real enforcement is server-side.

The Sales Pipeline

Leads progress through these statuses. When a lead reaches "Closed Won", it can be converted to a client:

Potential Prospect → Qualified Prospect → Interested Prospect → Hot Prospect → Hot Lead → Closed Won ✓ → Convert to Client
Lost ← can happen at any stage
09 — Security
Defense in Depth
🔒
HttpOnly Cookies
Session token is inaccessible to JavaScript — prevents XSS token theft.
🔑
bcrypt Password Hashing
Passwords stored as bcrypt hashes (cost factor 10). Plaintext never stored or logged.
🛡️
Rate Limiting
7 login attempts per 15 minutes per IP+username. In-memory map with automatic cleanup.
🧹
Input Validation
Server-side allowlists for status values and product categories. Rejects unknown values.
🔐
Service Role Key
Database accessed with service role key (server-side only). Never exposed to the browser.
👤
Ownership Checks
Every mutation verifies the record belongs to the requesting engineer (or admin). No cross-tenant access.
🚫
Generic Error Messages
"Invalid username or password" for both wrong-user and wrong-password to prevent enumeration.
📋
XSS Prevention
All user data escaped with esc() before innerHTML injection. Content-Type: nosniff header.
🌐
Security Headers
HSTS (1 year), X-Frame-Options: DENY, Referrer-Policy: strict-origin, X-Content-Type-Options: nosniff.
