Important context that shaped this pass. The live frontend authenticates against a separate, off-repo Supabase Edge Functions deployment (FUNCTIONS_BASE) and talks to it for almost everything — only /api/activities hits this repo. There are no repo equivalents for the Edge Functions' /subscribe (push) and /location endpoints. 🔑Auth unification (C4 fixed, C3 partial)

FixedC4 Single shared session helperlib/auth.js · api/activities.js

activities.js had its own Bearer-only getSession() while every other endpoint used the cookie-based one. I deleted the duplicate and pointed activities.js at the shared lib/auth.js helper. To avoid breaking the current Bearer-token frontend, getSession() now resolves the token from either the HttpOnly cookie (preferred) or an Authorization: Bearer header.

// lib/auth.js — one helper, both transports
function getTokenFromRequest(req){
  const cookies = parseCookies(req.headers.cookie || '');
  if (cookies[COOKIE_NAME]) return cookies[COOKIE_NAME];
  const h = req.headers['authorization'] || '';
  return h.startsWith('Bearer ') ? h.slice(7).trim() : null;
}

✓ Every endpoint now shares one auth path; no behavior change for the live frontend.

DeferredC3 Token still in localStorage (groundwork laid)index.html ↔ backend

The shared helper now accepts cookies, so the server side is ready. Finishing C3 means the frontend must stop storing the token in localStorage and rely on the HttpOnly cookie instead — but the live backend is a cross-origin Edge Functions deployment with no repo /subscribe or /location equivalents. Rewiring that blindly would break push, location, and possibly auth (cross-origin cookies).

Needs your decision: which backend is canonical — the repo's same-origin /api/*, or the Edge Functions? Once decided I can complete the migration (set cookie on login, switch all fetch to credentials:'include', remove getAuthToken/setAuthToken, port /subscribe + /location if moving to /api).

🟠High — all 5 fixed

FixedH1 Removed exposed DDL endpointapi/run-migration.js
Deleted the file. It ran CREATE TABLE at runtime via the Management API using the wrong credential and leaked upstream errors. The table DDL still lives in migrations/site_activities.sql for out-of-band use.
FixedH2 Require the service-role keylib/db.js
Dropped the silent || SUPABASE_ANON_KEY fallback. With RLS allow-all policies the API must use the service-role key; falling back to anon would silently change authorization. It now throws if the key is missing.
FixedH3 DB-backed login rate limitingapi/auth.js · migrations/login_attempts.sql
Replaced the per-instance in-memory Map + setInterval (useless across serverless instances/cold starts) with a shared login_attempts table. All checks fail open so a limiter outage can never lock out real users. Also switched the rate-limit key to Vercel's x-real-ip first (L12), reducing header spoofing.
FixedH4 CSP no longer blocks the mapvercel.json
Added https://unpkg.com to script-src/style-src/connect-src and https://*.tile.openstreetmap.org to img-src, so Leaflet and its tiles load when the Location feature is re-enabled. (Self-hosting Leaflet remains the stronger long-term option — noted below.)
FixedH5 No more raw error leakageapi/activities.js (+ all handlers)
activities.js returned err.message to the client; it now logs server-side and returns a generic 'Server error', matching the other endpoints. The setup_required signal is kept but no longer echoes the raw driver text.
